# Chess Persistence Benchmarks

This directory contains JMH benchmarks for comparing MongoDB and PostgreSQL persistence performance.

## Prerequisites

1. **Docker** - Required to run MongoDB and PostgreSQL containers
2. **SBT** - Scala Build Tool
3. **Java 17+** - Required for running the benchmarks

## Setup

### 1. Start Databases

From the project root directory, start the MongoDB and PostgreSQL containers:

```bash
docker compose up -d
```

Verify the databases are running:

```bash
docker compose ps
```

You should see:
- `chess-mongodb` running on port 27017
- `chess-postgres` running on port 5432

### 2. Verify Database Connectivity

Test MongoDB:
```bash
docker exec -it chess-mongodb mongosh -u chess -p chess123
```

Test PostgreSQL:
```bash
docker exec -it chess-postgres psql -U chess -d chess
```

## Running Benchmarks

### Run All Benchmarks

```bash
sbt "Benchmarks/Jmh/run"
```

### Run Specific Benchmark

Run only MongoDB benchmarks:
```bash
sbt "Benchmarks/Jmh/run .*mongo.*"
```

Run only PostgreSQL benchmarks:
```bash
sbt "Benchmarks/Jmh/run .*postgres.*"
```

Run specific operation (e.g., save operations):
```bash
sbt "Benchmarks/Jmh/run .*Save.*"
```

### Quick Test Run

For a faster test run with fewer iterations:
```bash
sbt "Benchmarks/Jmh/run -i 1 -wi 1 -f 1"
```

Where:
- `-i 1` = 1 measurement iteration (instead of 5)
- `-wi 1` = 1 warmup iteration (instead of 3)
- `-f 1` = 1 fork (already the default)

## Benchmark Operations

The benchmark suite tests the following operations:

### Write Operations
- **mongoSaveSingleGame / postgresSaveSingleGame** - Save a single game
- **mongoSaveBulk10 / postgresSaveBulk10** - Save 10 games sequentially

### Read Operations
- **mongoFindById / postgresFindById** - Retrieve a game by UUID
- **mongoFindAll100 / postgresFindAll100** - Retrieve 100 games with pagination
- **mongoFindByStatus / postgresFindByStatus** - Query games by status
- **mongoFindByOpening / postgresFindByOpening** - Query games by ECO opening code

## Understanding Results

JMH will output results in two modes:

1. **Throughput** (ops/ms) - Higher is better
   - How many operations per millisecond

2. **AverageTime** (ms/op) - Lower is better
   - Average time per operation in milliseconds

Example output:
```
Benchmark                                    Mode  Cnt   Score    Error   Units
DatabasePersistenceBenchmark.mongoSaveSingle thrpt    5  45.234 ±  2.103  ops/ms
DatabasePersistenceBenchmark.mongoSaveSingle  avgt    5   0.022 ±  0.001   ms/op
```

## Results Location

Benchmark results are written to:
- Console output (summary)
- `benchmark/results/` directory (detailed JSON results)

## Test Data

The benchmarks use realistic test data generated by `TestDataGenerator`:

- **Game size**: 10-80 moves per game (typical chess game length)
- **Data variety**: Mix of in-progress and completed games
- **Openings**: Common ECO-coded chess openings (A00-E99)
- **Pre-populated data**: 500 games in each database for read tests

## Customization

### Adjust Warmup/Measurement

Edit the annotations in `DatabasePersistenceBenchmark.scala`:

```scala
@Warmup(iterations = 3, time = 2, timeUnit = TimeUnit.SECONDS)
@Measurement(iterations = 5, time = 3, timeUnit = TimeUnit.SECONDS)
```

### Change Test Data Volume

Modify the setup method to adjust pre-populated data:

```scala
val prePopulateGames = TestDataGenerator.generateBatch(500, ...) // Change 500 to desired count
```

### Add New Benchmarks

Add methods with the `@Benchmark` annotation:

```scala
@Benchmark
def myCustomBenchmark(): Result =
  // Your benchmark code
```

## Cleanup

Stop and remove the database containers:

```bash
docker compose down
```

Remove volumes (WARNING: deletes all data):
```bash
docker compose down -v
```

## Educational Notes

These benchmarks demonstrate:

1. **Document vs Relational Storage**
   - MongoDB stores games as BSON documents
   - PostgreSQL uses relational tables with arrays for move history

2. **Indexing Strategies**
   - Both databases use indexes on id, status, openingEco, and createdAt
   - Index effectiveness varies by database type

3. **Query Performance Patterns**
   - Primary key lookups (findById) are fast in both
   - Range queries and filtering behave differently
   - Pagination performance varies with dataset size

4. **Write Patterns**
   - Upsert operations (insert or update)
   - Bulk write performance
   - Transaction overhead differences

## Troubleshooting

### Databases not running
```
Error: Connection refused
```
Solution: Start databases with `docker compose up -d`

### Port conflicts
```
Error: Port 27017 already in use
```
Solution: Stop existing MongoDB instance or change port in docker-compose.yml

### Out of memory
```
Error: Java heap space
```
Solution: Increase JVM memory in Fork annotation:
```scala
@Fork(value = 1, jvmArgs = Array("-Xms4G", "-Xmx4G")) // Increase from 2G to 4G
```
